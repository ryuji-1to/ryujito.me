# Design System

## 1. 位置づけ

このサイトは、派手なプロダクト紹介ではなく、個人の記録と文章を静かに読むための場所として設計する。
印象の中心は「読みやすさ」「落ち着き」「誠実さ」で、視覚表現はその補助に徹する。

実装を見ると、ルートでは `bg-main-bg` / `text-main-text` を全体の基調にし、本文領域は `max-w-2xl` の細いカラムで構成されている。
トップページ、記事一覧、記事詳細、自己紹介のどれも、情報量より読み心地を優先している。

このドキュメントは、既存実装をもとに次の判断を揃えるための基準とする。

- `src/styles.css` のトークンをどう使うか
- Tailwind ユーティリティをどの粒度で使い分けるか
- 新しいページやコンポーネントを既存のトーンにどう寄せるか

## 2. 全体の雰囲気

このサイトのビジュアルは、ニュートラルなグレースケールを土台にした、軽くミニマルなエディトリアル UI に近い。
ただし雑誌的に装飾するのではなく、個人サイトらしい近さを残す。

特徴は次の通り。

- 背景は真っ白ではなく、ほぼ白に近いごく薄いグレーで統一する
- 文字色も真っ黒ではなく、やわらかいダークグレーを使う
- 境界は強い線ではなく、背景差と薄いボーダーで作る
- アクセントカラーは常用せず、例外的な意味づけにだけ使う
- 見出しジャンプは控えめで、本文と見出しの距離が近い
- 余白で構造を作り、装飾で構造を作りすぎない

避けたい方向性:

- マーケティングサイト風の大きなヒーロー
- 彩度の高い配色
- 過剰なカード化
- 影に頼った立体表現
- 情報より UI が前に出る構図

## 3. 実装から見えるデザイン原則

既存コードから拾えるルールを先に言語化しておく。

### 3.1 細い 1 カラムを基本にする

`src/routes/__root.tsx` では本文領域を `max-w-2xl w-full mx-auto px-24 py-40 sm:px-40` で構成している。
つまり基本形は、中央寄せの細い 1 カラムと、左右に十分な余白を持つレイアウトである。

このサイトでは、横方向に情報を広げるより、縦方向に静かに積む方が合う。

### 3.2 余白の主役は 16 / 24 / 40

実装上よく出てくるのは `gap-16`、`p-24`、`space-y-40`、`py-40` である。
これは次の役割分担として読むと扱いやすい。

- `8`: 微調整、メタ情報、インライン要素
- `16`: 近い要素同士の間隔
- `24`: コンポーネント内部の標準余白
- `40`: セクション間の間隔

### 3.3 装飾より階層差で読ませる

`Timeline`、`Markdown`、`Posts` どれも、見出しサイズや背景差、境界線、余白差で情報を整理している。
色や影は強調ではなく補助として使われている。

### 3.4 ダークモードは必須の相方

ルート、ヘッダー、タイムライン、Markdown などで `dark:*` が対になっている。
新しい UI を足すときは、ライトだけで完結させない。

## 4. カラーパレット

このリポジトリでは `src/styles.css` に light / dark のグレースケールと、少数のアクセントカラーが定義されている。
パレットの性格は「無彩色中心、コントラストは十分、彩色は控えめ」である。

### 4.1 基本色

| 役割 | トークン | 値 | 用途 |
|---|---|---|---|
| ページ背景 | `--color-main-bg` | `var(--color-gray-1)` | ルート背景 |
| 補助背景 | `--color-sub-bg` | `var(--color-gray-2)` | カード、補助面 |
| 主本文 | `--color-main-text` | `var(--color-gray-12)` | 標準テキスト |
| 補助本文 | `--color-sub-text` | `var(--color-gray-11)` | 日付、説明、区切りの弱い文字 |
| ダーク背景 | `--color-dark-main-bg` | `var(--color-dark-gray-1)` | ダーク時のルート背景 |
| ダーク補助背景 | `--color-dark-sub-bg` | `var(--color-dark-gray-2)` | ダーク時のカード面 |
| ダーク主本文 | `--color-dark-main-text` | `var(--color-dark-gray-12)` | ダーク時の主文字 |
| ダーク補助本文 | `--color-dark-sub-text` | `var(--color-dark-gray-11)` | ダーク時の補助文字 |

### 4.2 グレースケールの役割

`gray-1` から `gray-12` は単純な濃淡ではなく、用途で切り分けると扱いやすい。

- `gray-1` / `gray-2`: 面の差を作る背景
- `gray-4` / `gray-5` / `gray-6`: ボーダー、コード背景、補助ブロック
- `gray-8`: 区切りや弱い視覚ノイズ
- `gray-11` / `gray-12`: 補助本文と主本文

ダーク側でも同じ考え方で、`dark-gray-1` と `dark-gray-2` が面、`dark-gray-4` から `dark-gray-6` が境界、`dark-gray-11` と `dark-gray-12` が文字である。

### 4.3 アクセントカラー

アクセントカラーは以下のみ。

- `red` / `dark-red`
- `orange` / `dark-orange`
- `green` / `dark-green`

既存実装では Markdown の疑似ブラウザ UI にだけ使われている。
つまりブランドカラーではなく、意味のある記号的な差し色として使うべきものである。

## 5. 色トークンの使い方

色は生の値で書かず、まず役割トークンから選ぶ。
特に新しいページでは、`main` / `sub` 系で十分なら `gray-*` を直接使わない。

### 5.1 優先順位

1. `bg-main-bg` / `text-main-text` など役割トークンを使う
2. 役割トークンで足りないときだけ `gray-*` を使う
3. ライトとダークの両方で繰り返し必要なら `src/styles.css` に役割トークンを追加する
4. 生の hex 値は原則書かない

### 5.2 よく使う組み合わせ

| 用途 | 推奨クラス |
|---|---|
| ページ全体 | `bg-main-bg text-main-text dark:bg-dark-main-bg dark:text-dark-main-text` |
| 補助面 | `bg-sub-bg dark:bg-dark-sub-bg` |
| 説明文 | `text-sub-text dark:text-dark-sub-text` |
| 控えめな線 | `border border-gray-6 dark:border-dark-gray-6` |
| 弱い区切り | `text-gray-8 dark:text-dark-gray-8` |
| インラインコード背景 | `bg-gray-4 dark:bg-dark-gray-4` |

### 5.3 実際の使われ方

- `Header`: 区切りの `/` に `text-gray-8 dark:text-dark-gray-8`
- `Timeline`: カード面に `bg-sub-bg dark:bg-dark-sub-bg`、枠線に `border-gray-6 dark:border-dark-gray-6`
- `Post detail`: 日付に `text-sub-text dark:text-dark-sub-text`
- `Footer`: 面の切り替えに `bg-gray-2 dark:bg-dark-gray-2`
- `Markdown`: カードやコード背景に `gray-4` から `gray-5`

### 5.4 やらないこと

- `gray-8` 以上の濃い色を広い面に敷く
- 装飾目的でアクセントカラーを常用する
- ライトだけにしか意味がない色指定を増やす
- コンポーネントごとに別の色体系を持ち込む

## 6. タイポグラフィ

### 6.1 フォント

`src/styles.css` では次の 2 系統が定義されている。

- `--font-sans`: システム UI 向けの sans-serif
- `--font-ud`: `BIZ UDGothic` を含む本文向けフォント

ルートでは `font-ud` が適用されているため、サイトの基本文体は UD ゴシック系である。
読みやすさと日本語の安定感を優先した判断になっている。

### 6.2 サイズスケール

| トークン | 値 | 役割の目安 |
|---|---|---|
| `text-xxs` | `0.667rem` | タイムライン説明、かなり小さい補足 |
| `text-xs` | `0.8rem` | 日付、メタ情報、小さな注釈 |
| `text-sm` | `0.887rem` | ナビゲーション、記事一覧のタイトル |
| `text-base` | `1rem` | 標準本文 |
| `text-md` | `1.1431rem` | 本文より少し強い導入文 |
| `text-lg` | `1.3331rem` | 小見出し候補 |
| `text-xl` | `1.6rem` | 記事タイトル |
| `text-2xl` | `2rem` | ページ見出し候補 |
| `text-3xl` | `2.6675rem` | かなり大きい見出し |
| `text-4xl` | `4rem` | 特別な場面のみ |

### 6.3 既存実装の基準

- `text-sm`: Header の現在地リンク、Profile の導線、記事一覧リンク
- `text-xs`: 記事日付、Markdown の small ブロック、一覧メタ
- `text-xxs`: Timeline の説明
- `text-xl`: 記事詳細のタイトル

ここから読むと、このサイトは大見出しを多用する設計ではない。
既存ページに追加するなら、まず `text-base`、`text-sm`、`text-xs`、必要なら `text-xl` までで考える。

### 6.4 行間と文章の扱い

`body` では `overflow-wrap: anywhere`、`word-break: normal`、`line-break: strict` が指定されている。
さらに `p { max-width: 80ch; }` により本文行長も抑えられている。

つまり文章の読み心地は、コンポーネント単位ではなく全体設計として守っている。
長文コンテンツではこれを崩さない。

## 7. 余白とサイズ

### 7.1 スペーシングトークン

`src/styles.css` の spacing は次の値で定義されている。

| トークン | 値 |
|---|---|
| `spacing-1` | `1px` |
| `spacing-2` | `2px` |
| `spacing-4` | `4px` |
| `spacing-8` | `8px` |
| `spacing-16` | `16px` |
| `spacing-24` | `24px` |
| `spacing-40` | `40px` |
| `spacing-64` | `64px` |
| `spacing-104` | `104px` |
| `spacing-168` | `168px` |

### 7.2 運用ルール

- `4`: ボタン内やごく近い要素
- `8`: 小さな区切り、メタ情報、微差
- `16`: 標準的な gap
- `24`: カード内部の余白
- `40`: セクション間
- `64` 以上: 大きなレイアウトの breathing space

### 7.3 実際によく使うクラス

| 用途 | 推奨クラス |
|---|---|
| 小さな gap | `gap-4`, `gap-8` |
| 標準 gap | `gap-16` |
| カード内余白 | `p-24` |
| ページ左右余白 | `px-24 sm:px-40` |
| ページ上下余白 | `py-40` |
| セクション積み上げ | `space-y-40` |
| 小さなまとまり | `space-y-16`, `space-y-24` |

### 7.4 例外値の扱い

既存コードには `w-[10px] h-[10px]` のような局所的例外もあるが、これは装飾点のような特殊用途に限定されている。
レイアウトの基準値として新しい任意値を増やさない。

## 8. 角丸と境界線

### 8.1 Radius

| トークン | 値 | 用途 |
|---|---|---|
| `rounded-2` | `2px` | かなり小さな補助用途 |
| `rounded-4` | `4px` | インラインコード、疑似ブラウザ上部 |
| `rounded-6` | `6px` | 標準カード |
| `rounded-8` | `8px` | 補助カード、強めの面 |
| `rounded-full` | `9999px` | アバター、タグ、小さな丸要素 |

### 8.2 運用ルール

- テキスト主体のカードは `rounded-6`
- 補助ブロックや callout は `rounded-8`
- インライン要素やコードは `rounded-4`
- 円である意味があるときだけ `rounded-full`

### 8.3 Border

既存実装では、ボーダーは `gray-4` から `gray-6` を中心にかなり薄く使う。
役割は「目立たせる」ことではなく、「面の存在を軽く示す」ことにある。

## 9. コンポーネント別ガイド

### 9.1 Root Layout

基準ファイル: `src/routes/__root.tsx`

- ルートで `font-ud` と light / dark の基調色を決める
- 本文は `max-w-2xl`
- 余白は `px-24 py-40 sm:px-40`
- Header と Footer は本文より前に出しすぎない

### 9.2 Header

基準ファイル: `src/components/header.tsx`

- sticky だが存在感は弱く保つ
- `backdrop-blur-xs` は補助的な効果として使う
- 現在地表示は小さな文字で十分
- 区切り記号は `gray-8` のような薄い色でよい

### 9.3 Profile

基準ファイル: `src/components/profile.tsx`

- 導入として簡潔であることが重要
- アバターは小さく、情報の主役は文章
- 導線は `text-sm` で軽く見せる

### 9.4 Timeline

基準ファイル: `src/components/timeline.tsx`

- 面は `sub-bg`
- 枠は `gray-6`
- 日付は `text-xs`
- 説明は `text-xxs`
- 装飾点や線はコンテンツを邪魔しない濃さで止める

このコンポーネントは「カードを使うが、カード UI には寄せすぎない」のよい基準になっている。

### 9.5 Footer

基準ファイル: `src/components/footer.tsx`

- 背景を `gray-2` に切り替えてページ終端を示す
- 情報量は少なく、呼吸を整える場所として使う
- アイコンサイズも主張しすぎない

### 9.6 Posts List

基準ファイル: `src/routes/posts/index.tsx`

- 一覧は `list-custom` と `space-y-16`
- タイトルは `text-sm`
- 日付は `text-xs text-sub-text font-mono`
- リンクの強調は hover underline 程度に留める

### 9.7 Post Detail

基準ファイル: `src/routes/posts/$slug.tsx`

- 記事タイトルは `text-xl`
- ヘッダーは `my-40 pb-8 space-y-40 border-b`
- タイトルと日付の間には `space-y-24`
- 本文に入る前の区切りは明確だが大げさにしない

### 9.8 Markdown

基準ファイル: `src/share/components/markdown.tsx`

- ベースは `prose prose-md dark:prose-invert`
- `prose-headings:text-balance` で見出しの崩れを抑える
- インラインコードは `text-xs`, `bg-gray-4`, `rounded-4`
- 補助カードは `bg-gray-2`, `border-gray-5`, `rounded-8`
- 疑似ブラウザ UI のみアクセントカラーを使用する

Markdown は、このサイトで最も「読み物」らしさが強い場所なので、UI 的な派手さを持ち込みすぎない。

### 9.9 Tag

基準ファイル: `src/share/components/tag.tsx`

- `bg-gray-4`
- `px-8 py-2`
- `rounded-full`

タグは小さなラベルであり、主役ではない。
存在感を出したいときも色より密度で調整する。

### 9.10 Spinner

基準ファイル: `src/share/components/spinner.tsx`

- 外周は `text-gray-4`
- 進行部分は `fill-gray-11`

ローディングも主張しすぎない。

## 10. ダークモードの方針

このサイトのダークモードは、ライトテーマを反転するというより、同じ情報構造を暗い環境に置き換える設計である。

ルール:

- 面の関係を維持する
- 主本文と補助本文の差を維持する
- ボーダーは見えすぎず、消えすぎない
- アクセント色は light / dark で別トークンを使う

新しい UI を作るときは、少なくとも次を確認する。

1. 背景と文字のコントラストが足りているか
2. カード面とページ面が分離して見えるか
3. `focus-visible` が十分見えるか
4. 補助文字が沈みすぎていないか

## 11. アクセシビリティ

既存実装では `*:focus-visible` に 3px の点線アウトラインが定義されている。
これはサイトのアクセシビリティ上かなり重要な要素なので、消さない。

基本ルール:

- フォーカスリングを無効化しない
- 色だけで意味を伝えない
- 小さい文字ほどコントラストを意識する
- 見出しレベルを見た目だけで決めない
- 画像の `alt` は用途ベースで付ける

## 12. 新しく UI を足すときの手順

### 12.1 まず考えること

1. これは本文か、補助情報か、操作 UI か
2. 既存ページのどれに近いか
3. `main/sub` トークンだけで表現できるか
4. 既存の余白スケールに収まるか

### 12.2 実装の優先順位

1. 既存の Tailwind ユーティリティで組む
2. 既存のテーマトークンを使う
3. 再利用価値があるなら `src/styles.css` にトークンを追加する
4. 最後の手段として局所的な任意値を使う

### 12.3 トークンを追加してよい条件

- 複数箇所で再利用される
- light / dark の両方で意味がある
- 既存スケールの延長として自然
- 用途名ではなく役割名で命名できる

## 13. Do / Don't

### Do

- `bg-main-bg` と `text-main-text` を基準に考える
- 補助面には `sub-bg` 系を使う
- 余白で構造を作る
- 小さな文字はメタ情報に限定する
- `dark:*` を最初からセットで考える
- Markdown の読み味を壊さない

### Don't

- 新しいブランドカラーを足す
- 強い影を多用する
- カードを何重にも入れ子にする
- 大見出しだけで階層を作ろうとする
- 任意値をレイアウト基準にする
- ライトモードだけで完成扱いにする

## 14. 要約

このサイトのデザインシステムは、派手な個性を前面に出すものではない。
代わりに、少数のグレートークン、安定した余白スケール、控えめな文字階層、そしてダークモードを含めた一貫性で成立している。

迷ったときの判断順はこれでよい。

1. 読みやすいか
2. 既存ページに並べて自然か
3. 色や装飾が情報より前に出ていないか
4. トークンとスケールの再利用で解決できているか
